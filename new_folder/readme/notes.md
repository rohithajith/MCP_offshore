# Development Notes

## Milestone 3 Notes (Operational Expansion)
Implementing specific and tightly encapsulated "Write" actions combined with specific "Simulation/Plan" capabilities acts as the final foundational brick for the Master Orchestrator. When we inevitably introduce an Agentic LLM Orchestrator on top of this backend, it will natively possess the bounds to test routing scenarios (transfer planning tools) internally *before* committing allocations (reserve_stock functions). This keeps your operational systems robust to model hallucinations and limits blast radius purely to controlled actions defined natively in your `Repository` structure.

## Milestone 4 Notes (Observability & Audit)
Building a durable `ActionAudit` repository natively separates toy "wrapper LLM bindings" from an enterprise-tier interface. A key aspect of introducing an Agentic Orchestrator later is acknowledging that agents hallucinate heavily. Instead of patching string patterns inside the SDK, our decorators gracefully suppress raw stack traces and catch explicit `BusinessRuleViolationErrors` inside the Service bindings themselves, logging explicit JSON string boundaries the LLM understands as "Oops, my parameters were rejected" while Operations Admins read exactly *what* string caused the rejection within `offshore_logistics.db`! This keeps the codebase incredibly mature and bulletproof. Using decorators is an optimal implementation mechanism since we only modified the business structure of the MCP tools natively via a single prefix layout without destroying the structural architecture mappings.
